# 최종 보고서

## 1. 프로그램 주제와 목적

내 프로그램의 주제: 사용자 맞춤형 음악 추천 알고리즘 시스템

이 프로그램을 만든 이유: 진로가 웹/앱 개발자라 웹이나 앱에서 흔히 사용하는 맞춤형 알고리즘 시스템을 파이썬으로 직접 구현하고 싶었다. 이 프로그램을 통해 사용자가 좋아하는 노래 장르와 노래를 듣는 목적, 그리고 장르와 목적 둘 중 무엇을 더 중요하게 여기는지를 고려하여 사용자에게 맞는 노래를 추천해주고자 한다.


---

## 2. 문제 분해 및 모델링

### 입력
- 사용자의 닉네임
- 선호하는 음악 장르(댄스/발라드/힙합/포크 중 하나)
- 듣는 목적(집중하기/분위기띄우기 중 하나)
- 장르와 목적의 중요도 백분율을 공백으로 구분한 수치 (예: 60 40)

### 처리
- 사용자가 입력한 선호 장르와 음악 데이터베이스의 장르를 비교하여 일치하는 곡에 기본 추천 점수를 부여한다.
- 사용자의 목적에 따라 음악의 목적 점수를 추출하고, 중요도를 가중치로 장르 점수와 목적 점수에 곱해 최종 추천 점수를 계산한다.
- 계산된 최종 추천 점수를 기준으로 전체 음악 목록을 내림차순으로 정렬한다.

### 출력
- 사용자의 입력 정보 확인 메시지
- 최종 추천 점수가 높은 순서대로 정렬된 음악 목록 TOP 3

---

## 3. 사용한 2차원 리스트 설명

2차원 리스트 이름: music_db

한 행의 의미: 등록되어 있는 음악 곡 1개에 대한 고유한 속성 및 점수 데이터 정보 세트

각 열의 의미:

| 열 번호 | 의미 | 실제 예시 |
|---:|---|---|
| 0번 열 | 노래 제목 | "소문의 낙원" |
| 1번 열 | 가수(그룹) 명 | "AKMU(악뮤)" |
| 2번 열 | 음악 장르 | "포크" |
| 3번 열 | 집중할 때 듣기 좋은지에 대한 점수 | 90 |
| 4번 열 | 분위기를 띄울 때 듣기 좋은지에 대한 점수 | 30 |

이 2차원 리스트를 사용한 이유: 음악 1곡은 제목, 가수, 장르, 수치 데이터(집중 점수, 분위기 띄우기 점수)와 같은 여러 개의 데이터를 가지고 있고, 여러 개의 음악을 리스트로 정리해야 각각의 데이터를 처리할 때 반복문을 통해 행 단위로 순회하면서 필요한 인덱스의 열 데이터를 편리하게 추출 및 제어할 수 있으므로 2차원 리스트를 사용했다.

## 4. 함수 설명

| 함수 이름 | 역할 | 입력값 | 반환값 또는 출력 |
|---|---|---|---|
| calculate_scores | 데이터베이스를 순회하며 사용자 장르 일치 여부와 목적이 무엇인지에 따라 장르점수와 목적 점수를 부여하고 장르와 목적의 중요도를 기반으로 가중치를 곱해 최종 점수를 계산 및 리스트화 | db (2차원 리스트), user_genre (선호장르), user_purpose (목적), genre_weight (장르 중요도 비율), purpose_weight (목적 중요도 비율) | scored_list ([제목, 가수, 최종 점수]가 한 행인 2차원 리스트) |
| sort_and_print | 입력받은 추천 데이터 리스트를 최종 점수 기준으로 정렬한 뒤 TOP 3 곡을 정돈된 규격에 맞춰 출력 | scored_list ([제목, 가수, 최종 점수]가 한 행인 2차원 리스트) | 최종 사용자 맟춤 음악 TOP 3 순서대로 출력 |
| main | 사용자로부터 입력을 차례로 받고 연산 및 출력 함수들을 호출하는 주 흐름 제어 | 사용자의 입력(닉네임, 선호 장르, 목적, 중요도)을 수신함 | 입력받은 값 확인하는 문구 출력, 계산 및 출력 함수 호출 |

가장 중요한 함수 하나를 골라 자세히 설명하세요.

함수 이름: calculate_scores

설명: 설명:
이 함수는 프로그램의 핵심 '추천 연산 알고리즘' 엔진입니다. 입력받은 음악 데이터베이스(db)를 반복문으로 한 곡씩 탐색하며 속성 정보를 파싱합니다.
장르 비교 조건문을 통해 사용자가 좋아하는 장르와 일치하면 기본값 100점을 부여하고, 다중 분기문(if-elif)으로 사용자의 목적 환경에 어울리는 점수(집중용 vs 분위기용)를 적절히 추출합니다.
이후 각각에 가중치 소수점 비율을 곱해 합산한 종합 추천 점수를 계산한 뒤 이를 반올림합니다. 특히 조건 일치도가 전무해 점수가 0점 이하인 데이터는 리스트에 추가하지 않는 if score > 0 예외 필터링을 거치며, 정제된 [곡명, 아티스트, 최종점수] 배열을 결과 리스트(scored_list)에 차곡차곡 누적한 뒤 반환합니다.

## 5. 조건문과 반복문 활용

### 조건문 사용 부분

어떤 조건을 검사했는가?
- 2차원 리스트 내부의 음악 장르와 사용자가 입력한 선호 장르가 일치하는가?
- 사용자가 입력한 목적이 '집중하기'와 '분위기띄우기' 중 무엇인가?
- 사용자의 선호에 따라 계산된 각 음악의 최종 점수가 0점 초과인가?

### 반복문 사용 부분

무엇을 반복 처리했는가?
데이터베이스 전수 조사: for music in db 반복문을 돌며 music_db 2차원 리스트에 적재되어 있는 모든 가요 데이터 행을 한 행씩 끝까지 추적합니다.
랭킹 목록 제한 출력: for i in range(min(3, len(scored_list))) 구조를 채용하여, 정렬을 마친 스코어 리스트에서 최종 점수가 가장 높은 상위 3곡만 추출해 인덱스 순서대로 1위, 2위, 3위를 차례로 포맷팅하여 화면에 순차적으로 터미널 출력합니다.


### 조건문과 반복문이 함께 작동한 부분
calculate_scores 함수 내부에서 for music in db 문이 돌며 12곡을 하나씩 훑어 나갈 때, 루프 내부에서 각 음악 개체마다 장르 일치를 검사하는 조건문(if), 사용자의 선택 목적에 맞는 점수 열을 구별해 추출하는 분기 조건문(if-elif), 최종 스코어가 유효한 데이터인지 필터링하는 조건문(if score > 0)이 순차적으로 맞물려 실행됩니다.

## 8. Gemini 활용 내용 요약

Gemini에게 도움받은 부분:
음원 데이터베이스 확장 기획: 댄스, 발라드, 힙합, 포크 4종류의 카테고리가 데이터 개수 편향 없이 골고루 매핑되고 음악 고유 속성에 맞는 수치 데이터를 지닌 12곡의 음악 2차원 리스트 세트를 수립하는 데 조언을 구함.

동적 가중치 기반 리스트 내림차순 정렬: 리스트의 내림차순 정렬 메커니즘을 통합 구성하여 최종 산출 점수가 제일 높은 순으로 스코어 리스트를 정돈하는 뼈대 가이드라인을 얻음.

논리 버그 디버깅 검증: 0점인 비추천 음악들이 순위에 난무하던 연산 구멍을 해결하고, 매칭 데이터가 적을 때 터지는 인덱스 초과 예외를 해결하기 위해 if score > 0: 및 min() 함수를 적용하는 설계 조언을 얻음.

Gemini가 제안했지만 내가 수정하거나 사용하지 않은 부분:
Gemini는 입력 예외들을 사전에 완전히 차단하는 완성도를 위해 프로그램에 이중 무한 반복문(while True)을 입히고 복잡한 에러 캐치 루틴(try-except)을 대거 이식한 템플릿 코드를 제안했습니다. 하지만 이러한 복잡한 제어 구조는 고등 정보 교육 과정의 범위에 어긋날뿐더러, 소스 코드 자체의 가독성을 헤치고 직관적인 구동 흐름을 약화시킬 수 있다고 판단하여 채택하지 않았습니다.

최종 코드에서 내가 직접 판단하거나 수정한 부분:
데이터베이스를 확장할 때 Gemini가 제시한 특징 수치 점수들을 기계적으로 복제하지 않고 제 주관적 음악 감상 지식을 기초로 85, 95 등 세부 점수를 직접 보완 기입했습니다.

프로그램의 구동 논리를 스스로 명확하게 짚고 가기 위해, 무거운 다중 while문 등의 제안은 생략하는 대신 에러의 가장 깊은 본질적 실마리였던 '0점 데이터 필터링(if score > 0:)'과 '인덱스 범위 바인딩(min())'이라는 심플하면서 강력한 기본기 조건 논리 위주로만 내 프로그램에 주체적으로 직접 엮어내어 코드를 안전하고 가볍게 완성시켰습니다.

## 9. 한계와 개선 방안

현재 프로그램의 한계:
현재는 사용자가 터미널 콘솔 화면에 장르를 한글로 입력할 때 띄어쓰기 한 칸이라도 나면 장르 인식을 실패합니다. 또한, 중요도를 공백으로 정확히 가르지 않고 입력하면 분할 에러(ValueError)가 나는 등 오타나 잘못된 조작을 했을 때 완벽히 걸러내지 못하는 투박한 문자열 파싱 상태라는 한계가 존재합니다.

다시 개선한다면 추가하고 싶은 기능:
사용자가 글자를 완벽히 입력하지 않고 '댄'이나 '힙'처럼 앞 글자 한 단어만 입력해도 문자열 내포 여부(in) 비교문을 통해 자동으로 완성하여 맞춤 장르를 자동 보정 매칭해주는 유연한 텍스트 전처리 패서(Parser) 기능을 추가해보고 싶습니다. 또한, 파이썬 내장 .sort()를 뛰어넘어 컴퓨터공학에서 중요하게 다루는 퀵 정렬(Quick Sort) 알고리즘을 소스 코드에 손수 직접 이중 리스트 구조에 맞춰 코딩하여 백엔드 처리 효율을 획기적으로 올리는 연습을 진행하고 싶습니다.
